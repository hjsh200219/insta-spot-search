# Harness Principles — insta-spot-search

하네스(에이전트 팀이 이 리포에서 안전·정확·효율적으로 일하게 하는 문서·규칙·게이트의
총합)의 성숙도를 채점하는 기준과, 그 채점을 지탱하는 Anthropic 하네스 운영 원칙.
성숙도 레벨 산정은 [maturity-framework.md](./maturity-framework.md), 점수대별 개선
행동은 [fix-catalog.md](./fix-catalog.md), 실행 이력은 [gc-history.md](./gc-history.md).

## 12-원칙 채점 기준

각 원칙을 0~4로 채점한다(0 없음 / 1 미흡 / 2 부분 / 3 양호 / 4 모범). 소규모 스킬
리포이므로 프론트엔드·DB·배포 관련 원칙은 "N/A"로 두고 가중치에서 제외한다.

| # | 원칙 | 이 리포에서 무엇을 보는가 |
|---|------|---------------------------|
| 1 | 진입점 명료성 | `AGENTS.md`가 존재하고 지도 역할을 하는가(핸드북화되지 않았는가) |
| 2 | 아키텍처 문서화 | `ARCHITECTURE.md`가 레이어·데이터흐름·의존방향을 실제 함수명으로 기술하는가 |
| 3 | 코드 컨벤션 명시 | stdlib 전용·subprocess 리스트 인자·`glob.escape`가 문서로 강제되는가 |
| 4 | 보안 표면 정리 | 키·쿠키·업로드 표면이 `docs/SECURITY.md`에 집약되고 알려진 위험이 등재되는가 |
| 5 | 품질 게이트 | preflight 스모크·단위 테스트 대상·exit code 검증이 정의되는가 |

| 7 | 기술부채 가시성 | 중복·위험 항목이 트래커에 위치·우선순위와 함께 등재되는가 |

## Anthropic 하네스 운영 원칙

### 회의적 평가 원칙 (Skeptical Evaluation)

평가자는 기본값을 "미흡(not proven)"으로 두고, **증거가 제시될 때만** 점수를 올린다.
"코드가 잘 짜여 보인다"는 인상은 점수 근거가 아니다. 이 리포에서는 "테스트가 있다"는
주장은 실제로 실행되는 테스트 파일과 통과 로그가 있어야 인정하고, "보안이 안전하다"는
주장은 SECURITY.md의 표면 목록과 대조해 반증을 시도한 뒤에만 받아들인다. 낙관적 자기
평가는 하네스 성숙도를 부풀려 정작 필요한 개선(항목 d: 실네트워크·실바이너리
e2e 미검증)을 가린다.

### 하네스 단순화 원칙 (Harness Simplification)

하네스의 각 구성요소(체크리스트·강제 규칙·자동화)는 **현재 모델의 한계에 대한 가정**을
인코딩한 것이다. 모델이 나아지면 어떤 방어벽은 불필요한 마찰이 되고, 어떤 규칙은
여전히 필수로 남는다. 그러므로 하네스는 늘리기만 하는 대상이 아니라 **주기적으로
재검증**해 걷어낼 것을 걷어내야 한다. 이 소규모 리포에서 특히 중요하다 — 소스가 3파일인데
문서가 비대해지면 "지도"가 "핸드북"이 되어 원래 목적을 배신한다. 문서를 추가할 때마다
"이게 지금도 실제 코드 동작을 반영하는가, 아니면 관성으로 남았는가"를 물어야 한다.

### Phase 독립 에이전트 원칙 (Context Reset > Compaction)

여러 페이즈로 나뉜 하네스 작업은 각 페이즈를 **깨끗한 컨텍스트로 시작한 독립 에이전트**가
맡는 것이, 하나의 긴 컨텍스트를 압축(compaction)하며 이어가는 것보다 낫다. 압축은 미묘한
사실(어떤 exit code가 무엇과 매핑되는지 같은)을 손실 압축해 후속 판단을 오염시킨다. 리셋은
그 오염을 원천 차단한다. 그래서 각 페이즈 산출물은 **다음 페이즈가 파일만 읽고 이어갈 수
있도록** 자기완결적으로 쓴다(이 문서들이 서로를 링크로만 참조하고 내용을 복제하지 않는 이유).

### Sprint Contract 원칙

각 작업 스프린트는 시작 전에 **무엇을 만들고 무엇을 만들지 않을지**를 명시한 계약으로
경계를 고정한다. 이 리포의 하네스 셋업이 "프론트엔드/DB 문서를 만들지 않는다",
"CLAUDE.md·harness-setup.md는 다른 페이즈 소관"을 명시하는 것이 그 예다. 계약이
스코프 크리프(빈 껍데기 문서 양산)를 막고, 완료 판정을 객관화한다. 산출물은 계약의
"만들 것" 목록과 1:1로 대조되어야 하며, 계약 밖 산출물은 그 자체로 결함이다.

### 채점 앵커 원칙 (Per-score Anchors)

점수 척도의 각 눈금(0/1/2/3/4)에 **구체적 관찰 기준(anchor)**을 붙여야 평가자 간·시점 간
드리프트를 막는다. "3점은 대체로 좋음" 같은 모호한 서술은 평가자마다 다르게 읽힌다.
이 리포에서는 예컨대 "테스트 자동화" 원칙에서 0=테스트 0개, 2=순수 함수 일부 커버,
4=순수 함수 전체 + exit code 계약 테스트 통과처럼 **셀 수 있는 앵커**로 고정한다.
maturity-framework.md의 레벨 정의가 이 앵커를 레벨 단위로 구현한 것이다.
